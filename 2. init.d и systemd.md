# 2. Работа с init.d и systemd: теория и практика

## Теоретическая часть

### 2.1. Что такое init.d, как он работает, как настроить работу сервисов

После загрузки ядра Linux запускается **первый процесс** — **init** (PID 1). В классической схеме **SysV init** управление сервисами строится на **скриптах** в каталоге **/etc/init.d/**.

**Как это устроено:**

- В `/etc/init.d/` лежат исполняемые скрипты. Каждый скрипт управляет одним сервисом и принимает аргумент: **start**, **stop**, **restart**, **status** (иногда **reload**). При вызове, например, `/etc/init.d/nginx start` скрипт запускает демон nginx; при `stop` — останавливает.
- **Уровни запуска (runlevels)** задаются номерами 0–6. Для каждого уровня есть каталог `/etc/rcN.d/` (например, `/etc/rc2.d/`). В нём лежат **симлинки** на скрипты из `/etc/init.d/`:
  - симлинки с префиксом **K** (Kill) — сервисы, которые нужно остановить при входе в этот уровень;
  - симлинки с префиксом **S** (Start) — сервисы, которые нужно запустить. Цифра в имени (S20nginx) задаёт порядок: сначала с меньшим номером.
- При загрузке системы init переходит на «обычный» уровень (например, 2 или 5); при этом по очереди выполняются скрипты из соответствующего `rcN.d/`: сначала все K*, затем все S* с аргументом start.

**Как настроить работу сервиса с init.d:**

1. **Добавить скрипт** в `/etc/init.d/` с поддержкой аргументов start/stop/restart/status (шаблон можно скопировать с существующего сервиса).
2. **Включить автозапуск** на нужных уровнях — создать симлинки в `/etc/rcN.d/`. В Debian/Ubuntu это делается командой:
   - `sudo update-rc.d имя_сервиса defaults` — добавить сервис в уровни по умолчанию (S-ссылки при старте, K-ссылки при остановке);
   - `sudo update-rc.d имя_сервиса remove` — убрать автозапуск.
3. **Запуск/остановка вручную:** `sudo /etc/init.d/имя_сервиса start|stop|restart|status`.

**Практика (если в системе есть init.d-скрипты):**

```bash
ls /etc/init.d/
sudo /etc/init.d/nginx status
sudo /etc/init.d/nginx start
# Включить автозапуск (Debian/Ubuntu)
sudo update-rc.d nginx defaults
# Убрать автозапуск
sudo update-rc.d nginx remove
```

На современных дистрибутивах большинство сервисов переведено на systemd; init.d часто остаётся для совместимости или отдельных пакетов.

### 2.2. Что такое systemd, как он работает, основы теории

**systemd** — современная система инициализации и менеджер сервисов. Она заменяет SysV init и выполняет роль PID 1: запускает сервисы, управляет зависимостями, перезапускает упавшие процессы, обрабатывает переключение целей загрузки.

**Основы теории:**

- **Unit (юнит)** — единица управления. Описывается конфигурационным файлом. Типы: **service** (демон), **socket** (сокет для активации по требованию), **target** (группа юнитов — аналог runlevel), **timer** (расписание), **path**, **mount** и др.
- **Unit-файлы** лежат в `/usr/lib/systemd/system/` (от пакетов) и в `/etc/systemd/system/` (переопределения и свои сервисы). Файлы из `/etc` имеют приоритет.
- **Зависимости:** в unit можно указать, после каких других unit'ов запускать (After=), что требуется для работы (Requires=, Wants=). systemd сам выстраивает порядок запуска и при старте цели (например, multi-user.target) поднимает все нужные сервисы.
- **Цели (targets)** — аналог runlevels: **multi-user.target** (многопользовательский без GUI), **graphical.target** (с GUI), **rescue.target** (восстановление) и т.д. Текущая цель по умолчанию задаётся симлинком `/etc/systemd/system/default.target`.

**Как управлять сервисами:**

- `systemctl start|stop|restart|reload имя` — запуск, остановка, перезапуск, перезагрузка конфига.
- `systemctl enable|disable имя` — включить/выключить автозапуск при загрузке текущей цели.
- `systemctl status имя` — статус unit'а (активен, неактивен, ошибка, логи).
- После изменения unit-файлов обязательно: `sudo systemctl daemon-reload`.

### 2.3. Как настроить сервис: пример настройки приложения в виде сервиса

Чтобы приложение запускалось при загрузке и управлялось через systemd, создают **unit-файл** типа **service**. Пример — простое приложение (например, скрипт или бинарник), которое должно работать в фоне и перезапускаться при падении.

**Шаг 1. Создать unit-файл**

Файл: `/etc/systemd/system/myapp.service`

```ini
[Unit]
Description=My application service
Documentation=https://example.com/docs
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=myapp
Group=myapp
ExecStart=/usr/bin/python3 /opt/myapp/app.py
Restart=on-failure
RestartSec=5
Environment=APP_CONFIG=/etc/myapp/config.yaml

[Install]
WantedBy=multi-user.target
```

**Расшифровка секций и опций:**

- **[Unit]**  
  - **Description** — краткое описание.  
  - **After=network-online.target** — запускать после того, как сеть «поднята».  
  - **Wants=network-online.target** — желательная зависимость от сети (без неё сервис тоже может стартовать, но порядок учтён).

- **[Service]**  
  - **Type=simple** — процесс, запущенный через ExecStart, считается основным (подходит для долгоживущего демона). Для программы, которая «отваливается» в фоне, иногда используют **Type=forking** и **PIDFile=**.  
  - **User/Group** — от чьего имени запускать (безопасность и права на файлы).  
  - **ExecStart** — команда запуска.  
  - **Restart=on-failure** — перезапускать при ненулевом коде выхода или сигнале.  
  - **RestartSec=5** — пауза перед перезапуском.  
  - **Environment** — переменные окружения для процесса.

- **[Install]**  
  - **WantedBy=multi-user.target** — при «включении» сервиса (enable) он будет запускаться при загрузке цели multi-user.target (типовой серверный режим).

**Шаг 2. Применить и запустить**

```bash
# Перечитать конфигурацию systemd
sudo systemctl daemon-reload

# Включить автозапуск и запустить сейчас
sudo systemctl enable --now myapp.service

# Проверить статус и логи
systemctl status myapp
journalctl -u myapp -f
```

**Шаг 3. Проверка при проблемах**

- Логи: `journalctl -u myapp -n 50` или `journalctl -u myapp -f`.  
- Если меняли unit: снова `daemon-reload`, затем `restart`.  
- Проверить путь в ExecStart, права на файлы и пользователя (User/Group).

**Практика — свой unit:**

1. Создайте пользователя (при необходимости): `sudo useradd -r -s /bin/false myapp`.  
2. Положите скрипт или бинарник, например, в `/opt/myapp/` и выставите права.  
3. Создайте `/etc/systemd/system/myapp.service` по примеру выше, подставив свои ExecStart, User, Group.  
4. Выполните `daemon-reload`, `enable --now myapp`, проверьте `status` и `journalctl -u myapp`.

### 2.3.1. Подробно о файлах .service: расположение, структура и параметры

Unit-файлы сервисов имеют расширение **.service** и лежат в одном из каталогов (по приоритету):

| Каталог | Назначение |
|---------|------------|
| **/etc/systemd/system/** | Локальные и переопределяющие unit'ы. **Сюда кладут свои** myapp.service и переопределения из пакетов. Имеет наивысший приоритет. |
| **/usr/lib/systemd/system/** | Unit'ы, поставляемые пакетами дистрибутива. Вручную править не рекомендуется — обновление пакета перезапишет файл. |
| **/run/systemd/system/** | Временные unit'ы (сбрасываются при перезагрузке). |

Имя файла задаёт имя unit'а: `myapp.service` → unit `myapp` (или `myapp.service`). Синтаксис — INI-подобный: секции в квадратных скобках `[Section]`, параметры `Ключ=Значение`. Пустые строки и строки, начинающиеся с `;` или `#`, игнорируются. Значения можно продолжать на следующей строке с отступом или использовать обратный слэш в конце строки.

Ниже — основные параметры по секциям и **когда их применять**.

---

#### Секция [Unit]

Общие метаданные и зависимости; применимы ко всем типам unit'ов (не только service).

| Параметр | Описание | Когда применять |
|----------|----------|------------------|
| **Description=** | Краткое описание сервиса (до одной строки). Показывается в `systemctl status` и в списках. | Всегда полезно задать для своих сервисов. |
| **Documentation=** | Ссылки на документацию (URL или пути к man). | По желанию, для справки. |
| **After=** | Список unit'ов; данный unit будет **запускаться после** них (только порядок, не зависимость). | Когда важен порядок: например, `After=network-online.target` — после поднятия сети; `After=nginx.service` — после nginx. |
| **Before=** | Данный unit должен **запуститься до** указанных. | Реже; обычно достаточно After в другом unit'е. |
| **Requires=** | Жёсткая зависимость: если указанный unit не активируется или падает, данный тоже останавливается. | Когда сервис **не может работать** без другого (например, приложение без БД). Осторожно: падение зависимости тянет за собой ваш сервис. |
| **Wants=** | Мягкая зависимость: «желательно запустить после». Падение другого unit'а **не** останавливает ваш. | Часто достаточно вместо Requires: `Wants=network-online.target` — запустить после сети, но не падать, если сеть потом отвалится. |
| **BindsTo=** | Как Requires, плюс остановка при остановке указанного unit'а. | Специфичные сценарии (привязка к устройству/монту). |
| **PartOf=** | Остановка указанного unit'а ведёт к остановке данного (обратная связь). | Когда сервис — «часть» другого (например, вспомогательный процесс). |
| **ConditionPathExists=**, **ConditionPathExists=!** | Запускать только если файл/каталог существует (или не существует). | Когда сервис имеет смысл только при наличии конфига или сокета. |
| **ConditionPathIsDirectory=** | Условие по каталогу. | Аналогично, для проверки каталогов. |

Типичная комбинация для сетевого приложения: `After=network-online.target`, `Wants=network-online.target` — запуск после сети без жёсткой привязки.

---

#### Секция [Service]

Параметры, специфичные для типа **service**.

**Тип сервиса (Type=):**

| Значение | Поведение | Когда применять |
|----------|-----------|------------------|
| **simple** | Процесс, запущенный в ExecStart, считается главным. systemd считает сервис «запущенным» сразу после старта процесса. | **По умолчанию.** Долгоживущий процесс (демон), который не отваливается в фоне. Типично для приложений на Python, Node, Go и т.п. |
| **forking** | Ожидается, что процесс из ExecStart сделает fork и завершится; главным считается дочерний процесс. systemd ждёт появления дочернего. | Классические демоны, которые делают `fork()` и родитель сразу выходит (nginx, Apache и др.). Обязательно указывать **PIDFile=**, чтобы systemd знал PID потомка. |
| **oneshot** | ExecStart выполняется один раз и завершается. Сервис считается активным, пока команда не завершится. С **RemainAfterExit=yes** сервис остаётся «активным» после завершения. | Разовые задачи при загрузке (монтирование, настройка сети), скрипты инициализации. |
| **notify** | Сервис должен отправить сообщение через sd_notify(), когда готов. systemd ждёт этого сообщения. | Демоны, поддерживающие systemd notify (часть современного софта). |
| **idle** | Запуск откладывается, пока не закончатся активные задания (редко нужно). | Специальные случаи. |
| **dbus** | Активация при появлении имени на D-Bus. | Сервисы, регистрирующиеся в D-Bus. |

**Запуск и остановка:**

| Параметр | Описание | Когда применять |
|----------|----------|------------------|
| **ExecStart=** | Команда запуска (полный путь к исполняемому файлу + аргументы). Обязателен для service. | Всегда. Использовать абсолютные пути. Для shell-команд: `ExecStart=/bin/bash -c '...'`. |
| **ExecStartPre=**, **ExecStartPost=** | Команды до и после ExecStart (по порядку). | Предзапуск: создать каталог, проверить конфиг. Постзапуск: логирование, уведомление. |
| **ExecStop=** | Команда остановки. По умолчанию systemd шлёт SIGTERM главному процессу. | Когда нужна корректная остановка (закрыть сокет, сохранить данные). Например, `ExecStop=/bin/kill -TERM $MAINPID`. |
| **ExecReload=** | Команда перезагрузки конфига (reload). По умолчанию — SIGHUP. | Когда приложение поддерживает reload без полного рестарта (nginx, многие демоны). |
| **TimeoutStartSec=**, **TimeoutStopSec=** | Таймаут на старт и остановку (секунды). По умолчанию 90 с (start). | Долгий старт (БД, тяжёлое приложение) — увеличить TimeoutStartSec. Долгая остановка — TimeoutStopSec. |
| **KillMode=** | Что убивать при остановке: **control-group** (вся группа), **process** (только главный), **mixed**, **none**. | По умолчанию control-group. Если у процесса есть дочерние, которые должны быть убиты вместе — оставить или mixed. |
| **KillSignal=** | Сигнал при остановке (по умолчанию SIGTERM). | Если приложение ждёт другой сигнал — указать, например **KillSignal=SIGINT**. |

**Перезапуск при сбое:**

| Параметр | Описание | Когда применять |
|----------|----------|------------------|
| **Restart=** | **no** — не перезапускать (по умолчанию). **on-failure** — при ненулевом коде выхода, падении по сигналу или таймауту. **on-abnormal** — при падении по сигналу/таймауту. **on-watchdog** — по watchdog. **always** — перезапускать при любом выходе. | Для демонов обычно **on-failure**. **always** — если процесс должен «вечно» крутиться (в т.ч. при нормальном exit 0). |
| **RestartSec=** | Пауза (секунды) перед перезапуском после падения. | Чтобы не дёргать приложение при кратковременных сбоях (например, 5–10 с). |
| **StartLimitIntervalSec=**, **StartLimitBurst=** | В течение интервала (сек) допускается не более N перезапусков; при превышении сервис переходит в failed. | Защита от бесконечного цикла перезапусков (например, 60 с и 5 попыток). |

**Пользователь, окружение, ресурсы:**

| Параметр | Описание | Когда применять |
|----------|----------|------------------|
| **User=**, **Group=** | От чьего имени запускать процесс. | **Всегда** для своих сервисов — не запускать от root без необходимости. Отдельный пользователь/группа для приложения (безопасность и права на файлы). |
| **WorkingDirectory=** | Рабочий каталог процесса. | Когда приложение ожидает запуск из определённой директории (относительные пути в конфиге, логи). |
| **Environment=** | Переменные окружения (можно несколько строк). **EnvironmentFile=** — файл с переменными (например, `/etc/default/myapp`). | Конфиг через переменные (порты, пути, ключи). Секреты лучше через EnvironmentFile с ограниченными правами. |
| **Nice=** | Приоритет планировщика (nice) от -20 до 19. | Понизить приоритет фонового сервиса (например, 5 или 10). |
| **LimitNOFILE=** | Лимит открытых файлов (дескрипторов). | Для сервисов с большим числом соединений (прокси, очереди). |
| **MemoryMax=**, **CPUQuota=** | Ограничение памяти и CPU (systemd resource control). | Когда нужно ограничить потребление ресурсов сервисом. |

**Специфичные сценарии:**

| Параметр | Описание | Когда применять |
|----------|----------|------------------|
| **PIDFile=** | Файл, в который главный процесс записывает свой PID. | **Обязательно** при **Type=forking** — иначе systemd не узнает, какой процесс главный. |
| **RemainAfterExit=** | При **Type=oneshot**: считать сервис активным и после завершения ExecStart. | Службы «одноразовой» настройки, которые должны считаться «включёнными» после выполнения (например, применение правил при загрузке). |
| **StandardOutput=**, **StandardError=** | Куда перенаправить stdout/stderr: **journal**, **syslog**, **kmsg**, **null**, **file:path**. | По умолчанию journal. Для отладки можно временно **file:/var/log/myapp.log**. |
| **SyslogIdentifier=** | Имя в логах (journal/syslog). По умолчанию — имя unit'а. | Чтобы в логах было удобное имя приложения. |

---

#### Секция [Install]

Используется только при **enable/disable** — привязка к цели (target).

| Параметр | Описание | Когда применять |
|----------|----------|------------------|
| **WantedBy=** | При `systemctl enable` создаётся симлинк в указанной цели (например, `multi-user.target`). Сервис будет запускаться при загрузке этой цели. | **Обычно один из:** **multi-user.target** (сервер без GUI), **graphical.target** (с GUI). |
| **RequiredBy=** | Аналог WantedBy, но жёсткая обратная зависимость цели от сервиса (редко нужен). | Специальные случаи. |
| **Also=** | Дополнительные unit'ы включать/выключать вместе с данным. | Когда у сервиса есть парный timer или socket — enable одного тянет за собой другой. |

---

#### Пример с комментариями (сводка параметров)

```ini
[Unit]
Description=My application
Documentation=https://example.com/docs
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=myapp
Group=myapp
WorkingDirectory=/opt/myapp
EnvironmentFile=/etc/default/myapp
ExecStart=/usr/bin/python3 /opt/myapp/app.py
ExecReload=/bin/kill -HUP $MAINPID
Restart=on-failure
RestartSec=10
TimeoutStartSec=30
StandardOutput=journal
SyslogIdentifier=myapp

[Install]
WantedBy=multi-user.target
```

После любых изменений unit-файла выполнять: `sudo systemctl daemon-reload`, затем `restart` или `start` при необходимости. Полный список параметров и детали: `man systemd.service`, `man systemd.unit`, `man systemd.exec`.

## Практические задания

**Задание 1. init.d**

1. Выполните `ls /etc/init.d/`. Выберите один скрипт (если есть) и выполните `sudo /etc/init.d/имя status`. Запишите вывод.  
2. Объясните своими словами: зачем в каталогах `/etc/rcN.d/` используются симлинки с префиксами K и S и в каком порядке они выполняются.

**Задание 2. systemd — основы**

1. Выполните `systemctl get-default`. Запишите текущую цель по умолчанию.  
2. Выполните `systemctl list-units --type=service --state=running | head -15`. Запишите три любых активных сервиса.  
3. Для одного из них выполните `systemctl status имя` и `systemctl is-enabled имя`. Запишите, активен ли сервис и включён ли автозапуск.

**Задание 3. Пример сервиса**

1. Создайте unit-файл `/etc/systemd/system/test-svc.service` с типом **oneshot**, который в ExecStart выполняет одну команду (например, `echo "OK" >> /tmp/test-svc.log`). В секции [Install] укажите `WantedBy=multi-user.target`.  
2. Выполните `sudo systemctl daemon-reload`, `sudo systemctl start test-svc.service`. Проверьте, что команда выполнилась (например, содержимое `/tmp/test-svc.log`).  
3. Сделайте сервис включаемым при загрузке: `sudo systemctl enable test-svc.service`. Проверьте: `systemctl is-enabled test-svc.service`. При необходимости отключите: `sudo systemctl disable test-svc.service`.
