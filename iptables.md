
# iptables

## Что такое iptables

**iptables** — это утилита в Linux для настройки правил фильтрации и преобразования сетевых пакетов на уровне ядра. С её помощью вы решаете: какие пакеты принять, какие отбросить, куда перенаправить. Это основа фаервола на многих серверах.

**Важно:** правила применяются сразу. Ошибочная настройка может заблокировать ваш SSH-доступ. Поэтому:
- Работайте по возможности в консоли (не только по SSH) или держите сессию открытой до проверки.
- Сначала разрешайте нужное (SSH, loopback, established), потом уже ужесточайте политику.
- Сохраняйте рабочие правила и знайте, как их восстановить.

---

## Таблицы и цепочки

### Таблицы (Tables)

iptables использует **таблицы** — наборы правил для разных задач. Для начала познакомимся с таблицей **filter**.

| Таблица   | Назначение                         | Как часто нужна |
|----------|-------------------------------------|----------------------|
| **filter** | Фильтрация (разрешить/запретить)   | Да, с неё и начинать |
| **nat**    | NAT (подмена адресов/портов)       | Позже, для шлюза/VPN |
| **mangle** | Изменение заголовков пакетов       | Редко |
| **raw**    | Обработка до учёта соединений      | Редко |

По умолчанию команды `iptables` работают с таблицей **filter**. Для другой таблицы указывают `-t nat` (или mangle, raw).

### Цепочки (Chains)

В таблице **filter** есть три основные цепочки:

| Цепочка   | Назначение |
|----------|------------|
| **INPUT**   | Пакеты, которые приходят **на этот хост** (на сервер). |
| **OUTPUT**  | Пакеты, которые **исходят с этого хоста**. |
| **FORWARD** | Пакеты, которые **проходят через хост** (маршрутизатор). На обычном сервере часто пустая. |

На первом этапе вы в основном будете добавлять правила в **INPUT** (кого пускать на сервер).

## Просмотр текущих правил

Перед изменениями всегда смотрите, что уже есть.

```bash
# Список правил в цепочке INPUT (таблица filter)
sudo iptables -L INPUT

# То же, но с номерами строк (удобно для удаления)
sudo iptables -L INPUT -n -v --line-numbers
```

Параметры:
- **-L** — list (показать правила).
- **-n** — не резолвить IP в имена (быстрее и нагляднее).
- **-v** — verbose (пакеты/байты).
- **--line-numbers** — номера строк.

Политика по умолчанию (что делать с пакетом, если ни одно правило не сработало):

```bash
sudo iptables -L -n -v
```

В начале строки будет что-то вроде `(policy ACCEPT)` или `(policy DROP)`.

## Как устроено правило

Общий вид:

```text
iptables -A <цепочка> [условия] -j <действие>
```

- **-A** — добавить правило **в конец** цепочки.
- **-I** — вставить **в начало** (например, `-I INPUT 1`).
- **-D** — удалить правило (по номеру или по копии правила).
- **-j** — jump, действие: ACCEPT, DROP, REJECT, LOG и т.д.

**Порядок важен:** первое совпавшее правило выполняется, остальные для этого пакета не проверяются. Поэтому «разрешающие» правила обычно ставят выше «запрещающих».

## Базовые условия (матчеры)

| Условие              | Пример                          | Значение |
|----------------------|----------------------------------|----------|
| Интерфейс            | `-i lo` или `-o eth0`           | Входящий / исходящий интерфейс |
| Протокол             | `-p tcp`, `-p udp`, `-p icmp`   | Тип протокола |
| Порт назначения      | `-p tcp --dport 22`             | Порт (для TCP/UDP) |
| Порт источника       | `--sport 1024:65535`            | Диапазон портов |
| IP источника         | `-s 192.168.1.100`              | Адрес отправителя |
| IP назначения        | `-d 10.0.0.5`                   | Адрес получателя |
| Подсеть              | `-s 192.168.1.0/24`             | CIDR |
| Состояние соединения | `-m state --state ESTABLISHED,RELATED` | Уже установленное или связанное соединение |

**-i** обычно используют только в цепочках INPUT, FORWARD, PREROUTING; **-o** — в OUTPUT, FORWARD, POSTROUTING.

---

## Действия (targets)

| Действие  | Кратко |
|-----------|--------|
| **ACCEPT** | Принять пакет. |
| **DROP**   | Отбросить без ответа (удалённая сторона будет ждать таймаут). |
| **REJECT** | Отклонить и отправить ответ (например, «port unreachable»). |
| **LOG**    | Записать в лог и продолжить проверку следующих правил (часто ставят перед DROP для отладки). |

Для начала достаточно ACCEPT, DROP и при необходимости LOG.

## Первые безопасные правила

Цель: не потерять доступ и разрешить минимум нужного.

### Разрешить loopback (обязательно)

```bash
sudo iptables -A INPUT -i lo -j ACCEPT
```

Интерфейс `lo` — локальный. Многие сервисы и сам SSH могут от него зависеть.

### Разрешить уже установленные соединения

```bash
sudo iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
```

Так вы разрешаете ответы на исходящие запросы (браузер, apt, ваш текущий SSH). Без этого после «закрытия всего» вы можете потерять сессию после первого же DROP по умолчанию.

### Разрешить SSH (порт 22)

```bash
sudo iptables -A INPUT -p tcp --dport 22 -j ACCEPT
```

Эти три правила (lo + ESTABLISHED,RELATED + SSH) обычно ставят **до** того, как менять политику по умолчанию на DROP.

### Политика по умолчанию

Сначала оставьте политику ACCEPT и проверьте, что правила работают:

```bash
sudo iptables -L INPUT -n -v --line-numbers
```

Когда убедитесь, что SSH и нужные порты разрешены, можно ужесточить:

```bash
sudo iptables -P INPUT DROP
sudo iptables -P FORWARD DROP
# OUTPUT часто оставляют ACCEPT, иначе сервер не сможет ходить в интернет
sudo iptables -P OUTPUT ACCEPT
```

**Порядок добавления для безопасности:**

1. `-A INPUT -i lo -j ACCEPT`
2. `-A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT`
3. `-A INPUT -p tcp --dport 22 -j ACCEPT`
4. При необходимости ещё порты (80, 443 и т.д.)
5. В конце: `-P INPUT DROP`

## Добавление, вставка и удаление правил

**Добавить в конец цепочки:**

```bash
sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT
```

**Вставить в начало (например, первым):**

```bash
sudo iptables -I INPUT 1 -p tcp --dport 443 -j ACCEPT
```

**Удалить по номеру строки** (номера смотрите в `-L --line-numbers`):

```bash
sudo iptables -D INPUT 3
```

**Удалить по точному совпадению правила:**

```bash
sudo iptables -D INPUT -p tcp --dport 80 -j ACCEPT
```

**Очистить все правила в цепочке** (политики не трогает):

```bash
sudo iptables -F INPUT
```

**Очистить все пользовательские цепочки и правила** (осторожно):

```bash
sudo iptables -F
sudo iptables -X
```

После этого политики (например, ACCEPT по умолчанию) останутся — трафик снова будет проходить. Используйте для быстрого «сброса» при эксперименте.

## Сохранение и восстановление правил

Правила iptables **не сохраняются** после перезагрузки, если не использовать специальные механизмы.

### Вариант 1: iptables-save / iptables-restore

```bash
# Сохранить в файл
sudo iptables-save | sudo tee /etc/iptables/rules.v4

# Восстановить
sudo iptables-restore < /etc/iptables/rules.v4
```

Чтобы правила поднимались при загрузке, нужен скрипт в systemd или init (см. ниже).

### Вариант 2: iptables-persistent (Debian/Ubuntu)

```bash
sudo apt update
sudo apt install iptables-persistent
```

При установке спросят, сохранить ли текущие правила — можно согласиться. Дальше:

```bash
# Сохранить текущие правила в файлы и при следующей загрузке они подхватятся
sudo netfilter-persistent save
```

Файлы обычно лежат в `/etc/iptables/`. После правок вручную снова выполните `sudo netfilter-persistent save`.

## Минимальная конфигурация «с нуля»

Пример последовательности для сервера с SSH и веб-сервером:

```bash
# Очистить правила (политики останутся)
sudo iptables -F
sudo iptables -X
# Политики по умолчанию
sudo iptables -P INPUT DROP
sudo iptables -P FORWARD DROP
sudo iptables -P OUTPUT ACCEPT
# Loopback
sudo iptables -A INPUT -i lo -j ACCEPT
# Уже установленные соединения
sudo iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
# SSH
sudo iptables -A INPUT -p tcp --dport 22 -j ACCEPT
# HTTP и HTTPS (по желанию)
sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 443 -j ACCEPT
# Сохранить (если установлен iptables-persistent)
sudo netfilter-persistent save
```

Проверьте SSH в другой сессии, прежде чем закрывать текущую.

## Ограничение доступа по IP

После того как базовые правила работают, можно ограничить доступ по **IP источника** (-s) и **назначения** (-d): разрешать SSH или другие сервисы только с определённой подсети или блокировать конкретные адреса.

### Зачем ограничивать по IP

- **SSH:** разрешить вход только с офисной сети или с одного адреса — снизить риск перебора паролей и брутфорса из интернета.
- **Служебные порты:** доступ к БД, панели администрирования только с доверенных хостов.
- **Чёрный список:** блокировать конкретные IP (сканеры, атакующие).

**Важно:** если вы заходите по SSH из сети с «плавающим» IP (домашний интернет без статики), ограничение по IP может вас самого выкинуть. Имеет смысл сначала разрешить SSH по ключу и отключить пароли, а ограничение по IP добавлять, когда есть стабильный диапазон (офис, VPN) или резервный способ доступа (консоль).

### Матчеры источника и назначения

В цепочке **INPUT** пакеты уже пришли на ваш хост, поэтому:
- **-s** — IP или подсеть источника (откуда пришло подключение), например `-s 192.168.1.100` или `-s 192.168.1.0/24`.
- **-d** — IP или подсеть назначения (куда пакет идёт); в INPUT часто не указывают.

### Разрешить SSH только с одной подсети

Разрешаем TCP порт 22 только для клиентов из подсети 192.168.1.0/24 (например, офис). Все остальные на порт 22 будут отброшены последующими правилами или политикой DROP.

```bash
# Сначала базовые правила: lo, ESTABLISHED,RELATED
sudo iptables -A INPUT -i lo -j ACCEPT
sudo iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
# SSH только из 192.168.1.0/24
sudo iptables -A INPUT -p tcp --dport 22 -s 192.168.1.0/24 -j ACCEPT
# Опционально: логировать и отклонить остальные попытки на SSH
sudo iptables -A INPUT -p tcp --dport 22 -j LOG --log-prefix "IPTables-SSH-DENY: "
sudo iptables -A INPUT -p tcp --dport 22 -j REJECT --reject-with tcp-reset
# Дальше другие разрешающие правила и в конце политика DROP
sudo iptables -P INPUT DROP
```

Порядок важен: правило «SSH из 192.168.1.0/24 — ACCEPT» должно быть **выше** правила «SSH — LOG/REJECT».

### Разрешить SSH с одного IP

```bash
sudo iptables -A INPUT -p tcp --dport 22 -s 203.0.113.50 -j ACCEPT
```

Доступ на порт 22 будет только с 203.0.113.50. Остальные на 22 можно явно REJECT/DROP (как в примере выше).

### Разрешить несколько подсетей или IP

Добавьте отдельное правило для каждой подсети:

```bash
# SSH из офиса и из VPN
sudo iptables -A INPUT -p tcp --dport 22 -s 192.168.1.0/24 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 22 -s 10.8.0.0/24 -j ACCEPT
```

В iptables нет «списка адресов в одном правиле» — либо несколько правил, либо отдельная цепочка.

### Запретить конкретный IP (чёрный список)

Запретить все входящие с адреса 198.51.100.1 (например, после скан-атаки). Правило DROP по источнику ставят **в начало** цепочки INPUT (или после lo и ESTABLISHED), чтобы остальные правила для этого IP не проверялись:

```bash
sudo iptables -I INPUT 3 -s 198.51.100.1 -j DROP
```

Номер 3 подберите под свою конфигурацию (после lo и ESTABLISHED,RELATED). Или вставьте в начало:

```bash
sudo iptables -I INPUT 1 -s 198.51.100.1 -j DROP
```

Проверка: `sudo iptables -L INPUT -n -v --line-numbers`.

### Ограничение по IP для других сервисов

Тот же принцип: правило с **-s** перед **-j ACCEPT**. Остальной трафик на этот порт обработают следующие правила или политика по умолчанию.

Пример: MySQL (3306) только с внутренней подсети:

```bash
sudo iptables -A INPUT -p tcp --dport 3306 -s 10.0.0.0/8 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 3306 -j DROP
```

Пример: панель администратора на порту 8080 только с одного IP:

```bash
sudo iptables -A INPUT -p tcp --dport 8080 -s 192.168.1.100 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 8080 -j DROP
```

### Просмотр и удаление правил с IP

```bash
# Список правил с номерами
sudo iptables -L INPUT -n -v --line-numbers

# Удалить правило по номеру (например, правило 4)
sudo iptables -D INPUT 4

# Удалить по точному совпадению
sudo iptables -D INPUT -p tcp --dport 22 -s 192.168.1.0/24 -j ACCEPT
```

После изменений не забудьте сохранить правила (`sudo netfilter-persistent save` или `iptables-save`).

**Кратко:** разрешающие правила с **-s** ставьте **выше** общих REJECT/DROP для того же порта; для SSH сначала убедитесь, что не отрежете себя; чёрный список (DROP по IP) удобно ставить в начало цепочки (**-I INPUT 1**); используйте подсети в формате CIDR: **-s 192.168.1.0/24**.

## Полезные команды для отладки

```bash
# Все правила с номерами и счётчиками
sudo iptables -L -n -v --line-numbers

# Только цепочка INPUT
sudo iptables -L INPUT -n -v --line-numbers

# Таблица nat (если настраивали NAT)
sudo iptables -t nat -L -n -v
```

При проблемах с доступом проверьте:
1. Есть ли правило, разрешающее ваш порт (22 для SSH).
2. Выше ли оно правил, которые всё запрещают.
3. Не стоит ли политика INPUT DROP без разрешающих правил перед ней.

## Задания для самостоятельной отработки

Выполняйте на **тестовой виртуальной машине** или сервере, к которому есть доступ по консоли. Не меняйте фаервол на рабочей системе без резервного доступа.

**1. Просмотр и понимание правил**

- Выведите текущие правила цепочки INPUT с номерами строк и без резолва имён (`-L INPUT -n -v --line-numbers`).
- Запишите: сколько правил в цепочке, какая политика по умолчанию (ACCEPT или DROP), есть ли правило для порта 22 (SSH).
- Кратко объясните, что делает каждое правило (одним предложением).

**2. Базовая конфигурация «с нуля»**

- Очистите все правила в таблице filter (`-F`, `-X`), не трогая политики.
- Добавьте по порядку: разрешить loopback, разрешить ESTABLISHED,RELATED, разрешить TCP порт 22 (SSH).
- Установите политику INPUT на DROP. Перед этим откройте вторую сессию по SSH и убедитесь, что подключение не рвётся.
- Выведите правила и сохраните их (`netfilter-persistent save` или `iptables-save` в файл). Проверьте, что после перезагрузки (или после `iptables-restore`) конфигурация восстанавливается.

**3. Добавление и удаление правил**

- Добавьте в конец цепочки INPUT правило: разрешить TCP порт 80 (HTTP). Проверьте вывод `iptables -L INPUT -n -v --line-numbers`.
- Вставьте в **начало** цепочки INPUT (номер 1) правило: разрешить TCP порт 443 (HTTPS). Объясните, почему порядок правил важен.
- Удалите одно из добавленных правил по номеру строки. Снова выведите правила и убедитесь, что нужное правило исчезло.

**4. Ограничение по IP (опционально)**

- Добавьте правило: разрешить SSH (порт 22) **только** с одной подсети (например, 192.168.1.0/24 — подставьте свою тестовую подсеть). Выше этого правила оставьте lo и ESTABLISHED,RELATED; ниже добавьте явный REJECT или DROP для порта 22, чтобы остальные попытки на SSH блокировались.
- Проверьте список правил. Убедитесь, что вы не отрезали себя от сервера (если заходите по SSH из другой подсети, сначала протестируйте из разрешённой или через консоль).
- Запишите команды, которые вы использовали, и кратко опишите, что получилось.
