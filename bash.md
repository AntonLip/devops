
# 1. Понятие Bash-скрипта

Bash (Bourne Again Shell) — это и интерактивная оболочка, под которой вы работаете в терминале, и интерпретатор языка скриптов. **Bash-скрипт** — обычный текстовый файл, в котором записана последовательность команд так, как вы бы вводили их в терминале, плюс конструкции языка: переменные, условия, циклы. Такой файл не компилируется; его по строкам читает и выполняет интерпретатор bash.

Скрипты нужны, чтобы не повторять одни и те же действия вручную: разовые задачи (сбор логов, проверка места на диске), периодические (бэкап по расписанию), сценарии развёртывания и администрирования. Всё, что можно сделать в командной строке, можно оформить в скрипт и вызывать одной командой.

В начале файла указывают **shebang** (hash-bang) — строку с путём к интерпретатору, чтобы система знала, какой программой запускать файл:

```bash
#!/bin/bash
# или, если bash может быть в разном месте:
#!/usr/bin/env bash
```

После shebang идут комментарии (строки, начинающиеся с `#`) и команды. Простейший пример:

```bash
#!/bin/bash
# Вывод приветствия
echo "Скрипт запущен."
echo "Текущий каталог: $(pwd)"
```

Команда `echo` выводит текст в стандартный вывод (stdout). Подстановка `$(pwd)` подставляет результат выполнения команды `pwd` в строку — это command substitution (подстановка вывода команды).

**Практика (демонстрация сразу после блока).** Создайте файл `report.sh`: shebang `#!/usr/bin/env bash`, комментарий «Отчёт по каталогу», две строки `echo "Отчёт по каталогу"` и `echo "Каталог: $1"`. Запустите `bash report.sh /tmp` и покажите вывод в консоли. Подробные шаги — в [[3.7 Практика bash#Практика к разделу 1]].

# 2. Запуск скриптов и переменные окружения

## Запуск

Скрипт можно выполнить двумя способами. Первый — сделать файл исполняемым и вызвать по пути:

```bash
chmod +x script.sh
./script.sh
```

Тогда ядро прочитает shebang из первой строки и запустит указанный интерпретатор, передав ему путь к файлу. Второй способ — явно указать интерпретатор; тогда не обязательно выставлять права на выполнение:

```bash
bash script.sh
```

В обоих случаях скрипт выполняется в **дочернем** процессе shell: переменные, которые вы меняете внутри скрипта, не изменяют окружение вашего текущего терминала (если только скрипт не вызывается через `source script.sh` или `. script.sh`, что выполняет его в текущем shell).

## Переменные окружения

Среда выполнения процесса складывается из **переменных окружения**, унаследованных от родительского процесса (логин-шелл, терминал), плюс переменных, заданных в скрипте. Типичные переменные:

| Переменная | Смысл |
|------------|--------|
| `PATH` | Список каталогов, в которых ищутся исполняемые файлы |
| `HOME` | Домашний каталог пользователя |
| `USER` | Имя текущего пользователя |
| `PWD` | Текущий рабочий каталог |
| `SHELL` | Путь к оболочке по умолчанию |

Присвоение в Bash делается без пробелов вокруг =; обращение к переменной — через `$NAME` или `${NAME}`.
Кавычки важны при пробелах и подстановках: `echo "$FILE"` выведет значение, а без кавычек слово может разбиться на несколько аргументов.

Чтобы переменная была видна не только в скрипте, но и в дочерних процессах (например, в программах, запускаемых из скрипта), её экспортируют:

```bash
export MY_VAR="value"
# или
MY_VAR="value"
export MY_VAR
```

## Позиционные параметры и код возврата

Аргументы командной строки доступны в скрипте как позиционные параметры:

| Обозначение   | Значение                                                              |
| ------------- | --------------------------------------------------------------------- |
| `$0`          | Имя скрипта (как он был вызван)                                       |
| `$1`, `$2`, … | Первый, второй и т.д. аргумент                                        |
| `$#`          | Количество аргументов                                                 |
| `$@`          | Все аргументы как отдельные слова (удобно для перебора)               |
| `$*`          | Все аргументы в одной строке                                          |
| `$?`          | Код возврата последней выполненной команды (0 — успех, не 0 — ошибка) |

Пример значения по умолчанию: если первый аргумент не передан, подставить `root`:

```bash
USER="${1:-root}"
DIR="${2:-/tmp}"
echo "User: $USER, Dir: $DIR"
```

Синтаксис `${1:-root}` означает: использовать `$1`, а если он пустой или не задан — использовать `root`.

**Практика (демонстрация сразу после блока).** В `report.sh` добавьте переменную `DIR="${1:-.}"` и переменные `LOG=/tmp/report.log`, `ERR=/tmp/report.err`. Выведите в консоль `USER`, `PWD`, `DIR` и при желании `$#`. Запустите `bash report.sh` и `bash report.sh /var/log` — сравните значение `DIR`. Подробные шаги — в [[3.7 Практика bash#Практика к разделу 2]].

# 3. Перенаправления: stdin, stdout, stderr

У каждого процесса по умолчанию есть три стандартных потока: **stdin** (стандартный ввод, файловый дескриптор 0), **stdout** (стандартный вывод, 1) и **stderr** (стандартный вывод ошибок, 2). Обычно stdin — это клавиатура, stdout и stderr выводятся в терминал. Перенаправления меняют, откуда процесс читает и куда пишет.

## Вывод в файл

- `command > file` — записать stdout в файл; файл перезаписывается. Если файл не существовал — создаётся.
- `command >> file` — дописать stdout в конец файла.
- `command 2> file` — записать stderr в файл (2 — номер дескриптора stderr).
- `command 2>> file` — дописать stderr в конец файла.

Чтобы и обычный вывод, и ошибки попадали в один файл (например, лог скрипта), используют объединение потоков:

```bash
./script.sh >> /var/log/myscript.log 2>&1
```

Здесь `>> /var/log/myscript.log` направляет stdout в файл, а `2>&1` — «stderr направить туда же, куда сейчас идёт stdout». Порядок важен: сначала задаётся куда идёт stdout, затем `2>&1` привязывает stderr к тому же месту.

Чтобы отбросить вывод (и не засорять консоль, и не создавать файл), используют устройство **/dev/null**:

```bash
command > /dev/null 2>&1
```

## Ввод из файла

- `command < file` — читать stdin из файла вместо клавиатуры. Например, цикл `while read line; do ... done < file` читает файл построчно.

## Пайпы

Конвейер (pipe) соединяет stdout одной команды с stdin другой:

```bash
cat /var/log/syslog | grep "error"
ls -la | tail -5
```

Первая команда пишет в stdout, вторая читает из stdin. stderr при этом по-прежнему идёт в терминал; чтобы перенаправить и его, можно писать `command 2>&1 | other`.

## Работа с несколькими дескрипторами

Когда нужно раздельно писать обычный вывод и ошибки в разные файлы или читать из нескольких источников, используют явные номера дескрипторов. В Bash можно открывать дополнительные дескрипторы (обычно 3–9) и привязывать их к файлам.

### Раздельная запись stdout и stderr в разные файлы

Частая задача — вести лог работы в один файл, а ошибки в другой, чтобы потом удобно анализировать сбои:

```bash
./script.sh > /var/log/myscript.log 2> /var/log/myscript.err
```

Если при этом нужно ещё и видеть вывод в терминале, сначала сохраняют текущий stdout в запасной дескриптор, затем дублируют вывод через `tee`:

```bash
# stdout — в файл и в консоль, stderr — только в файл ошибок
./script.sh 2> /var/log/myscript.err | tee -a /var/log/myscript.log
```

### Сохранение и восстановление дескрипторов

Чтобы временно перенаправить stdout в файл, а потом вернуть его «как было», используют копию дескриптора. Синтаксис `N>&M` означает: дескриптор N сделать копией M. Типичный приём — сохранить stdout в 3, перенаправить 1 в файл, выполнить команды, затем вернуть 1 из 3:

```bash
exec 3>&1                    # сохранить текущий stdout в дескриптор 3
exec 1>>/var/log/out.log     # stdout теперь в файл
echo "Это попадёт в out.log"
exec 1>&3                    # вернуть stdout из копии (дескриптор 3)
exec 3>&-                    # закрыть дескриптор 3
echo "Это снова в терминал"
```

То же можно сделать в одной строке для одной команды без `exec`: сначала открыть файл на запись в дескриптор 3, потом перенаправить 1 и 2:

```bash
# Одна команда: stdout в out.log, stderr в err.log
{ some_command 2>&3 1>out.log; } 3>err.log
```

Здесь `3>err.log` открывает err.log на запись в дескриптор 3; внутри блока `2>&3` отправляет stderr в этот файл, а `1>out.log` — stdout в другой файл.

### Чтение из нескольких источников

Дополнительный дескриптор удобен, когда нужно по очереди читать из двух файлов (например, основной список и список исключений). Открываем второй файл на чтение в дескриптор 3:

```bash
while IFS= read -r line; do
    echo "Обработка: $line"
done < list.txt

while IFS= read -r line <&3; do
    echo "Исключение: $line"
done 3< exclude.txt
```

`3< exclude.txt` открывает файл на чтение в дескриптор 3; `read -r line <&3` читает строку из этого дескриптора.

### exec и постоянное перенаправление

Команда **exec** без имени программы меняет перенаправления текущего shell (или скрипта) на всё оставшееся время выполнения. Поэтому в скриптах часто пишут:

```bash
exec 1>>/var/log/myscript.log 2>&1
```

После этой строки весь последующий вывод скрипта (и stdout, и stderr) идёт в файл. Если нужно часть вывода писать в другой файл, используют сохранение дескриптора (см. выше).

### Краткая сводка по синтаксису

| Запись | Значение |
|--------|----------|
| `N>file` | запись в file через дескриптор N |
| `N>>file` | дописывать в file через N |
| `N<file` | чтение из file через N |
| `N>&M` | дескриптор N сделать копией M (то же место вывода/ввода) |
| `N<&M` | то же для ввода |
| `N>&-` | закрыть дескриптор N |

Порядок имеет значение: система обрабатывает перенаправления слева направо, поэтому сначала задают «куда идёт stdout», потом «куда stderr» и т.д.

**Практика (демонстрация сразу после блока).** Запустите `report.sh` с перенаправлением: `bash report.sh /tmp >> /tmp/report.log 2>&1`, затем откройте лог. Либо добавьте в скрипт `exec >> "$LOG" 2>&1` после задания LOG. Вариант сложнее: раздельные файлы — при вызове `>> report.log 2> report.err`.

# 4. Условия, массивы, циклы

## 4.1. Условия

Проверки в Bash выполняются командой **test** или её синонимом — квадратными скобками `[ ]`. Пробелы вокруг скобок и внутри обязательны.

Проверки строк и файлов:

```bash
[ "$a" = "$b" ]     # строки равны
[ "$a" != "$b" ]    # не равны
[ -z "$a" ]         # строка пустая
[ -n "$a" ]         # строка непустая

[ -f "$path" ]      # путь — обычный файл
[ -d "$path" ]      # каталог
[ -r "$path" ]      # читаемость
[ -e "$path" ]      # существует
```

Числовые сравнения (внутри `[ ]` используют операторы с дефисом):

```bash
[ 5 -eq 5 ]   # равно
[ 5 -ne 3 ]   # не равно
[ 5 -lt 10 ]  # меньше
[ 5 -le 5 ]   # меньше или равно
[ 10 -gt 5 ]  # больше
[ 5 -ge 5 ]   # больше или равно
```

Условный оператор **if**:

```bash
if [ -f "$1" ]; then
    echo "Файл существует: $1"
elif [ -d "$1" ]; then
    echo "Это каталог: $1"
else
    echo "Не найден: $1"
fi
```

Выбор по значению переменной — **case**:

```bash
case "$1" in
    start)
        echo "Запуск..."
        ;;
    stop)
        echo "Остановка..."
        ;;
    *)
        echo "Использование: $0 {start|stop}"
        exit 1
        ;;
esac
```

## 4.2. Массивы

В Bash можно объявлять массивы с произвольной индексацией (индексы с 0). Объявление и обращение:

```bash
# Индексированный массив
arr=(one two three "four with spaces")

echo "${arr[0]}"        # первый элемент: one
echo "${arr[@]}"       # все элементы как отдельные слова (для перебора)
echo "${arr[*]}"       # все элементы в одной строке
echo "${#arr[@]}"      # количество элементов
```

Перебор элементов в цикле:

```bash
for item in "${arr[@]}"; do
    echo "Элемент: $item"
done
```

Добавление элемента: `arr+=(newitem)`.

#### Ассоциативные массивы (ключ–значение)

В **Bash 4 и выше** поддерживаются ассоциативные массивы — структура «ключ → значение», как словарь или хэш. Индексом служит произвольная строка, а не только число.

**Объявление** — обязательно через `declare -A` (без него обращение `map[key]=...` создаст обычный массив):

```bash
declare -A map
map[user]="alice"
map[level]="INFO"
map[ip]="192.168.0.10"
```

**Обращение:** по ключу — `${map[key]}`. Ключи при присвоении можно брать в кавычки, если содержат пробелы или спецсимволы: `map["some key"]="value"`.

**Перебор:** по ключам — `${!map[@]}` (восклицательный знак даёт список индексов); по значениям — `${map[@]}`. Пример вывода всех пар ключ–значение:

```bash
for key in "${!map[@]}"; do
    echo "  $key = ${map[$key]}"
done
```

**Проверка наличия ключа** в Bash 4.3+ — через `-v`:

```bash
if [[ -v map[user] ]]; then
    echo "Ключ user задан: ${map[user]}"
fi
```

**Пример использования** — накопление данных по ключу и вывод:

```bash
declare -A cnt
cnt[INFO]=0
cnt[WARN]=0
cnt[ERROR]=0

# Симулируем подсчёт из лога: встретили уровень — увеличили счётчик
for level in INFO WARN INFO ERROR INFO; do
    (( cnt[$level]++ ))
done

echo "Сводка по уровням:"
for k in INFO WARN ERROR; do
    echo "  $k: ${cnt[$k]}"
done
```

**Пример из жизни** — подсчёт запросов по коду ответа из access-лога nginx (девятое поле — HTTP-код). Ассоциативный массив удобен, когда ключи заранее неизвестны или их много:

```bash
declare -A by_code

while read -r line; do
    code=$(echo "$line" | awk '{print $9}')
    [[ -z "$code" ]] && continue
    (( by_code[$code]++ ))
done < /var/log/nginx/access.log

echo "Запросов по коду ответа:"
for code in "${!by_code[@]}"; do
    printf "  %s: %d\n" "$code" "${by_code[$code]}"
done
```

В результате получится вывод вида: `200: 15000`, `404: 320`, `500: 5` — типичная задача при разборе логов и отчётах по доступности сервиса.

На системах со старым Bash (например, macOS по умолчанию — Bash 3.2) ассоциативные массивы недоступны. Проверить версию: `bash --version`. Для переносимых скриптов можно эмулировать «словарь» через обычные переменные с составным именем (например, `map_user`, `map_level`) или вызывать внешние утилиты (awk, jq).

## 4.3. Циклы

Цикл **for** по списку слов:

```bash
for item in a b c; do
    echo "$item"
done
```

Перебор аргументов скрипта:

```bash
for arg in "$@"; do
    echo "Аргумент: $arg"
done
```

Цикл **while** выполняется, пока условие истинно:

```bash
while [ -f /tmp/lock ]; do
    echo "Ожидание снятия блокировки..."
    sleep 1
done
```

Чтение файла построчно:

```bash
while IFS= read -r line; do
    echo "Строка: $line"
done < /path/to/file
```

Объединение команд: `cmd1 && cmd2` выполнит cmd2 только если cmd1 завершилась успешно (код 0); `cmd1 || cmd2` — выполнит cmd2 только при неуспехе cmd1.
